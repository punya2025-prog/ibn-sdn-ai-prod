PART 3 — START IBN V2
================================================================

3.1 One-command start
──────────────────────
bash start_ibn_v2.sh start

# This script does all of the following automatically:
#  1. Loads .env
#  2. Activates venv
#  3. Checks/starts OLLAMA
#  4. Checks/starts Batfish
#  5. Deploys ContainerLab (13 cEOS switches)
#  6. Sets up management bridge 10.201.0.1/24
#  7. Enables eAPI on all 13 cEOS switches
#  8. Verifies eAPI reachability
#  9. Checks/builds RAG index
# 10. Starts IBN gateway on port 8001
# 11. Starts console HTTP server on port 9091
# 12. Runs health check

3.2 Manual start (step by step alternative)
─────────────────────────────────────────────
cd ~/ibn-sdn-ai-v2
source venv/bin/activate
export $(cat .env | grep -v '^#' | xargs)

# Step A: Deploy ContainerLab
sudo containerlab deploy \
  --topo topology/ibn-digital-twin-extended.clab.yml

# Step B: Set bridge IP
CLAB_BR=$(brctl show | awk 'NR>1 && $1~/^br-/{print $1}' | tail -1)
sudo ip addr add 10.201.0.1/24 dev $CLAB_BR 2>/dev/null || true

# Step C: Enable eAPI on all cEOS switches
for NODE in dc-spine1 dc-spine2 dc-leaf1 dc-leaf2 dc-leaf3 dc-leaf4 \
            campus-core1 campus-dist1 campus-dist2 \
            campus-access1 campus-access2 campus-access3 campus-access4; do
  docker exec clab-ibn-digital-twin-ext-$NODE Cli -c \
    "enable
configure
management api http-commands
   protocol http
   no shutdown
!
end" 2>/dev/null && echo "$NODE: eAPI OK" || echo "$NODE: SKIP"
done

# Step D: Start IBN gateway (background)
nohup python3 main.py > logs/gateway.log 2>&1 &
echo "Gateway PID: $!"

# Step E: Start console server (background)
nohup python3 -m http.server 9091 > logs/console.log 2>&1 &
echo "Console PID: $!"

# Step F: Verify
sleep 5
curl -s http://localhost:8001/health | python3 -m json.tool

3.3 Verify everything is running
──────────────────────────────────
bash start_ibn_v2.sh status

# Expected output:
# [OK]  Gateway PID=XXXXX  healthy  intents=0  rag=on
# [OK]  Console: running (port 9091)
# [OK]  OLLAMA: running
# [OK]  Batfish: running
# [OK]  ContainerLab: 13 cEOS containers running

PART 4 — IBN CONSOLE (ibn_console.html) — DETAILED GUIDE
================================================================

4.1 Access
───────────
Open in browser:
  http://192.168.20.15:9091/ibn_console.html

When it loads:
  Top bar shows: Gateway URL input box
  Type: http://192.168.20.15:8001
  Click: Connect
  Status shows: ● Connected  ● Healthy

4.2 Tab 1 — Topology Viewer
─────────────────────────────
Shows the 13-node interactive network diagram.

  - Click any switch node → popup shows IP, role, VLANs
  - Spine nodes (navy blue) at top
  - Leaf nodes (teal) in second row
  - Campus layer (amber) on right side
  - Access switches (gray) at bottom
  - WiFi AP nodes (green) below access switches
  - CCTV nodes (red) below access switches
  - Links shown as lines — bold=active, dashed=WiFi

  Buttons:
  [Refresh]  → re-polls topology API

  Click "Ask AI ↗" on a node popup → opens AI chat
  with that switch pre-selected for analysis.

4.3 Tab 2 — Host Configurator
───────────────────────────────
PURPOSE: Assign MAC/IP/VLAN to specific switch ports
         and verify reachability between two hosts.

Step-by-step usage:

  HOST A form (left side):
  1. MAC address → type or click [Random MACs]
     Example: ab:cd:ef:11:4c:01
  2. IP address  → match the VLAN subnet
     Example: 10.10.1.50  (VLAN 100 Finance)
  3. VLAN        → select from dropdown
     Example: 100 — Finance
  4. Switch      → select target switch
     Example: campus-access1
  5. Port        → click a port in the grid below
     OR type manually: Ethernet1

  PORT GRID colours:
  ┌─────────────────────────────────────────────────┐
  │ GREEN  = WiFi access point ports (Et21-26)      │
  │ RED    = CCTV camera ports (Et37-40)            │
  │ AMBER  = Uplink ports (Et49-52)                 │
  │ BLUE   = Currently selected port                │
  │ GRAY   = Available access ports                 │
  └─────────────────────────────────────────────────┘

  PORT GRID port numbers:
  Et1-12:  Finance workstations (VLAN 100)
  Et13-20: HR workstations (VLAN 200)
  Et21-24: WiFi Corporate APs (VLAN 1800)
  Et25-28: WiFi Guest APs (VLAN 1900)
  Et29-36: IP phones (VLAN 800)
  Et37-40: CCTV cameras (VLAN 1100)
  Et41-44: Printers (VLAN 1000)
  Et45-48: Guest kiosks (VLAN 400)
  Et49-52: Uplinks (do not use for hosts)

  "Assigning to:" toggle → click [Switch to Host B]
  to reassign port grid clicks to Host B.

  HOST B form (right side):
  Same fields — fill in the second host details.

  Buttons:
  [Configure hosts] → sends both hosts to IBN gateway
                      via POST /api/events/ingest
                      injects ARP/MAC entries to cEOS switches
  [Random MACs]     → generates realistic random MACs for both
  [Clear]           → resets log and host fields

  After clicking [Configure hosts]:
  Log shows:
    [10:24:01] Host A injected: 10.10.1.50 (ab:cd:ef:11:4c:01) VLAN 100
    [10:24:01] Host B injected: 10.10.2.50 VLAN 200
    [10:24:03] Run reachability test to verify

4.4 Tab 3 — Reachability Test
───────────────────────────────
PURPOSE: Test if two hosts can reach each other,
         using IBN simulation pipeline.

  Source IP:   pre-filled from Host A (or edit manually)
  Dest IP:     pre-filled from Host B
  Protocol:    ICMP / TCP:80 / TCP:443 / TCP:22 / UDP:53
               TCP:5432 (PostgreSQL) / TCP:3306 (MySQL)

  [Load from hosts] → fills IPs from Tab 2 Host A and B
  [Run test ↗]      → submits reachability intent to gateway
                      runs all 5 simulation gates
                      returns PASS or FAIL with explanation

  Result format:
    [10:25:01] PASS (342ms) — 10.10.1.50→10.10.2.50 reachable via routing
    [10:25:15] FAIL (180ms) — 10.10.11.1→10.10.12.1 blocked [PCI-DSS deny]

  [Run policy suite ↗] → runs all 8 pre-built policy tests:
    ✓ IoT → Finance          should DENY  (VLAN 300→100)
    ✓ Finance → Production   should ALLOW (VLAN 100→600)
    ✓ CCTV → Billing         should DENY  (VLAN 1100→1200 PCI-DSS)
    ✓ WiFi-Guest → Corporate should DENY  (VLAN 1900→any)
    ✓ App → DB               should ALLOW (VLAN 1500→1400)
    ✓ SCADA → IT             should DENY  (VLAN 700→any IEC-62443)
    ✓ WiFi-Corp → Dev        should ALLOW (VLAN 1800→500)
    ✓ Printers → Finance     should DENY  (VLAN 1000→100)

  Stats panel shows: Tests / Passed / Failed / Last RTT

4.5 Tab 4 — Production Changes
────────────────────────────────
Shows checklist of physical switch changes needed for production:
  - Arista 7010: enable NETCONF/gNMI/LLDP
  - Cisco 2960-X: enable OpenFlow → ODL
  - Security: change passwords, HTTPS, API keys
  - WiFi: WPA3-Enterprise, RADIUS, SSID separation

  [Full checklist ↗] → asks AI for complete production checklist
  [Arista commands ↗]→ asks AI for Arista 7010 commands
  [RADIUS config ↗]  → asks AI for FreeRADIUS 802.1X config

4.6 Left panel — Event Generator
──────────────────────────────────
  Intent text box:
  Type any natural language intent:
    "block IoT VLAN 300 from corporate"
    "prioritise VoIP traffic DSCP EF"
    "isolate CCTV from billing servers"
    "allow app servers to reach database on port 5432"

  [□ simulate] checkbox:
    checked   = run all 5 gates, do NOT deploy
    unchecked = simulate AND deploy if score ≥ 0.90

  [Submit ↗] → POST /api/channel/0/prompt
               result appears in event log within 4s

  Scenario dropdown:
    link_down, bgp_flap, arp_flood, rogue_ap,
    port_err, scada_isolate, voip_priority
  [Run scenario] → triggers pre-built event for selected type

  Auto-random:
  [● ON] / [● OFF] with interval (30s, 60s, 120s)
  When ON → gateway auto-generates realistic random intents
  Useful for demo and stress testing

4.7 Centre panel — Event Log
──────────────────────────────
Real-time scrolling log of all IBN events.

  Colour coding:
    GREEN  = DEPLOYED successfully
    AMBER  = PENDING approval / moderate score
    RED    = FAILED / rejected
    TEAL   = APPROVED by operator
    GRAY   = INFO / system messages

  Each event shows:
    Timestamp / State / Intent ID (first 8 chars)
    Score / Deployment method / Switch count

  Controls:
    [↑ Pause]    → freeze scroll (keeps receiving events)
    [↓ Resume]   → resume live scroll
    [▼ Bottom]   → jump to most recent event

4.8 Right panel — AI Chat
───────────────────────────
  Free-form question box:
  Examples:
    > what is the blast radius if dc-leaf2 goes down?
    > show all intents deployed in the last hour
    > why was intent b7c2 rejected?
    > create VLAN 300 isolation policy for IoT devices
    > what switches carry VLAN 1100?

  [APPROVE intent XXX] button:
  Appears when a PENDING approval intent is in the log.
  Click → sends approval to gateway → auto-deploys.

  [REJECT intent XXX] button:
  Click → marks intent as REJECTED → archived.

================================================================

  Port: 8002 (gateway)  9092 (console)
  TLS:  HTTPS on port 8443 (optional)
  Auth: API key required for all /api/ endpoints
  DB:   SQLite persistent intent store
  User: ibn-operator (privilege 5) on all switches
  Audit: /var/log/ibn/audit.log + syslog

7.1 Install V3
───────────────
unzip ibn-sdn-ai-v3.zip -d ~/
cd ~/ibn-sdn-ai-v3

# Copy .env from V2 and change port
cp ~/ibn-sdn-ai-v2/.env .env
sed -i 's/PORT=8001/PORT=8002/' .env
sed -i 's/9091/9092/'           .env

python3.12 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

7.2 Additional V3 setup
────────────────────────
# Create ibn-operator user on Arista 7010
ssh admin@192.168.20.105
  configure
  username ibn-operator privilege 5 role network-operator \
    secret STRONG_PASSWORD_HERE
  end
  write memory
  exit

# Enable NETCONF + gNMI + HTTPS eAPI on Arista 7010
ssh admin@192.168.20.105
  configure
  management api netconf
     transport ssh default
     no shutdown
  !
  management api gnmi
     transport grpc default
     port 6030
     no shutdown
  !
  management api http-commands
     protocol https
     no protocol http
     no shutdown
  !
  end
  write memory
  exit

# Verify ports open
for PORT in 22 80 443 830 6030; do
  timeout 2 bash -c "echo >/dev/tcp/192.168.20.105/$PORT" 2>/dev/null \
    && echo "port $PORT: OPEN" || echo "port $PORT: CLOSED"
done
# Expected after: 22 443 830 6030 OPEN  (80 CLOSED for prod)

PART 9 — TROUBLESHOOTING
================================================================

Problem: Gateway not starting
──────────────────────────────
tail -50 ~/ibn-sdn-ai-v2/logs/gateway.log
# Check for: port already in use, missing .env, import errors

Problem: cEOS eAPI not responding
───────────────────────────────────
# Check containers
docker ps | grep clab
# Check specific container
docker exec clab-ibn-digital-twin-ext-dc-leaf1 Cli \
  -c "show management api http-commands"
# Re-enable if needed
docker exec clab-ibn-digital-twin-ext-dc-leaf1 Cli -c \
  "enable
configure
management api http-commands
   protocol http
   no shutdown
end"

Problem: BGP inactive on campus/access switches
─────────────────────────────────────────────────
EXPECTED — campus and access switches do NOT run BGP.
Use gnn_telemetry_fix.py which handles this correctly.
BGP runs only on: dc-spine1, dc-spine2, dc-leaf1/2/3/4

Problem: RAG returning empty context
──────────────────────────────────────
# Rebuild RAG index
cd ~/ibn-sdn-ai-v2
source venv/bin/activate
python3 netbox_seed.py --mode offline --output network_state
python3 rag_indexer.py --rebuild

Problem: OLLAMA timeout on large intents
─────────────────────────────────────────
# Increase timeout in .env:
LLM_TIMEOUT=180
# Or switch to smaller model:
OLLAMA_MODEL=mistral
ollama pull mistral

Problem: ContainerLab deploy fails
────────────────────────────────────
# Check cEOS image exists
docker images | grep ceos
# Check containerlab version
containerlab version
# Try destroy and redeploy
sudo containerlab destroy --topo topology/ibn-digital-twin-extended.clab.yml
sudo containerlab deploy  --topo topology/ibn-digital-twin-extended.clab.yml

Problem: Arista 7010 NETCONF port 830 CLOSED
─────────────────────────────────────────────
SSH to Arista and run:
  configure
  management api netconf
     transport ssh default
     no shutdown
  end
  write memory
Then verify: nmap -p 830 192.168.20.105
